# Обзор: архитектура, скорость, размер

Оценка воксельного движка laiue с сохранением модульности. Код уже
сильно оптимизирован (бинарный greedy meshing, vertex pulling, GPU-пул
с суб-аллокацией и отложенным освобождением, кеш высот, дельты мира,
origin rebasing, сборка без CRT). Ниже — только предметные точки роста,
по приоритету.

## Сделано в этой сессии

- **Починен обрыв `chunk_streaming.c`** — прошлая сессия оборвала
  `ChunkStreamingDraw` на середине, проект не собирался. Хвост функции
  восстановлен.
- **Убрано мерцание чанков при правке блока.** Причина: `Invalidate`
  сразу удалял меш (`entry->mesh = NULL`), а рисовались только `READY`,
  поэтому чанк исчезал на кадр-два. Теперь старый меш держится и рисуется,
  пока рабочий поток строит новый; свап и освобождение — в `Pump`, когда
  замена загружена.
- **Закрыты швы между полигонами (T-стыки greedy-меша) без MSAA** —
  через консервативную растеризацию (см. ниже «О швах»).

## Скорость

1. **`WorldFillRegion`: аллокация буфера высот на каждую сборку меша**
   (`src/world/world.c`). `HeapAlloc`/`HeapFree` ~17 КБ (66×66 float)
   на каждый чанк, на рабочих потоках → churn аллокатора под нагрузкой
   стриминга. Решение: переиспользовать буфер из `ChunkMesherScratch`
   (по одному на поток), передав его в `WorldFillRegion` параметром.
   Вызывающий один — `BuildChunkMesh`. Эффект: средний, риск: низкий.

2. **`ChunkStreamingPump`: полный проход по таблице каждый кадр.**
   Шаг «повторные заявки» сканирует все `capacity` записей (при радиусе 5
   это ~4096) даже когда повторять нечего. Решение: счётчик записей без
   заявки или явный список «нужна заявка», чтобы пропускать проход, когда
   он пуст. Эффект: низкий-средний, риск: низкий.

3. **Линейный поиск по дельтам** (`ChunkGetDelta`/`ChunkSetDelta`).
   O(дельт) на блок. Обычно дельт мало — не проблема; при тяжёлом
   редактировании перейти на маленькую хеш-таблицу `localIndex`.
   Эффект: низкий, риск: низкий.

## Архитектура

1. **Двойной shared-lock в `WorldFillRegion`.** Проверка наличия дельт и
   их наложение — два отдельных прохода под `tableLock` с окном между
   ними. Свести к одному: собрать указатели пересекающих чанков один раз
   под замком. Убирает лишние блокировки и микро-гонку (сейчас она
   безопасна — стриминг всё равно ре-мешит при инвалидации). Риск: низкий.

2. **Модульность.** Непрозрачные хендлы, DLL на модуль, нет глобальных
   переменных, рендерер не знает о мире, мир — о геометрии. Претензий нет,
   разбивать дальше не нужно.

3. **Мелочь: захардкожены цвет блока (пиксельный шейдер) и цвет неба
   (clear color, `renderer_d3d12.c`).** Если планируются биомы/палитры —
   вынести в константы/конфиг. Организационное, не срочно.

## Размер

Уже очень плотно: `/NODEFAULTLIB`, `/NOENTRY`, `/OPT:REF /OPT:ICF`,
`/MERGE` секций, LTCG (MSVC) / thinLTO (clang), свои `memset`/`memcpy`.
Значимых сокращений не вижу. Закоммиченные заголовки шейдеров
(`chunk_vs.h`/`chunk_ps.h`) — это норма: сборка работает и без `fxc`.

## О швах (почему так, а не «один меш»)

Швы — это T-стыки, врождённые для любого greedy meshing (включая
vercidium): длинная кромка большого квада граничит с меньшими квадами,
чья общая вершина лежит на её середине; растеризатор округляет две кромки
чуть по-разному → субпиксельная щель, сквозь неё виден фон. Такие стыки
есть и внутри чанка, и на границах.

«Объединить всё в один меш» убрало бы только граничные стыки, но не
внутренние (их большинство), и сломало бы стриминг (правка блока →
перестройка гигантского меша). Поэтому выбрана консервативная
растеризация: GPU покрывает пиксель, если треугольник задел его хоть
краем, — соседние квады смыкаются, фон не просвечивает. Даром по памяти,
дешевле MSAA, требует GPU с Tier 1+ (все современные), иначе тихий откат
к обычной растеризации. Побочно: силуэты толще на ≤1px.

Полностью геометрический вариант (устранение T-стыков в мешере) —
водонепроницаем и без завязки на железо, но усложняет мешер и добавляет
треугольники (против цели по размеру/скорости). Оставлен как опция.
