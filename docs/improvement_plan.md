# План улучшений laiue

Актуально для версии 0.5.0. План основан на текущем дереве исходников и
успешной Release-сборке `ninja-clang-release`.

## Отложено — автоматические тесты

1. Подключить CTest и добавить небольшие тестовые EXE без CRT для чистых
   функций: `infinite_coord`, `terrain_noise`, `content_format`,
   `voxel_raycast`, `player_jump` и `player_stance`.
2. Добавить интеграционные тесты форматов: повреждённый `world.meta`,
   обратная совместимость `player.dat`, порядок `mods/enabled.txt`,
   несовместимая версия API и отсутствующая DLL.
3. В CI собирать Debug/Release как минимум clang-cl; отдельно запускать
   тесты и проверять, что репозиторий не меняется после генерации шейдеров.

Раздел сознательно отложен. Возвращаться к нему стоит перед изменением
ABI SDK или бинарных форматов версии 2.

## P1 — уменьшить стоимость изменений

1. После профилирования разделить `renderer_d3d12.c` на устройство и swap
   chain, загрузку ресурсов, chunk pass, panorama pass и UI pass. Общие
   D3D12-владельцы остаются внутри непрозрачного `Renderer`.
2. Разделить `world_infinite.c` на хранение чанков/дельт, генерацию и
   публичный фасад мира; сохранить текущий `world.h` как границу ABI.
3. Разнести `chunk_streaming.c` на планировщик, очередь работ и владение
   GPU-мешами. Явно описать переходы состояний чанка и проверки revision.
4. Разделить `pause_menu.c` на состояние/команды и представление. Это
   позволит тестировать настройки без окна и D3D12.
5. Убрать прямые относительные включения SDK из `core`: предоставить
   include-directory через CMake и включать `laiue_mod_api.h` по имени.

Критерий готовности: крупные подсистемы имеют файлы до примерно 800 строк,
а их внутренние части не становятся новыми DLL без реальной необходимости.

Механическое разбиение без покрытия или измеримого эффекта не считается
улучшением. Новые файлы остаются внутри существующих DLL.

## P1 — измерить и оптимизировать горячие пути

1. [Частично выполнено] Добавлены счётчики meshing, очередей, отменённых и
   отброшенных результатов, upload и среднего времени сборки. Остались
   GPU timing и вывод снимка в диагностический UI.
2. Снять профиль движения камеры по новому миру и по миру с дельтами;
   оптимизировать только подтверждённые узкие места.
3. [Частично выполнено] Не начатые заявки старого центра отменяются;
   upload ограничен 2 мс и четырьмя мешами за кадр. Остались приоритет
   ближних чанков и кооперативная отмена уже начатого meshing.
4. Для рендера измерять draw calls, объём upload, занятость общих GPU-
   буферов и число граней панорамы. Добавить предупреждения о thrashing.
5. Проверить аллокации и линейные поиски в `mods`, `content_catalog` и
   сохранениях; эти пути оптимизировать лишь если они попадают в кадр.

Критерий готовности: для каждой оптимизации есть базовое измерение,
целевой показатель и повторяемая сцена.

## Что пока не делать

- Не объединять DLL только ради уменьшения их числа: сначала измерить
  старт, размер и стоимость границ.
- Не добавлять ECS, job-system общего назначения или скриптовый runtime
  без нескольких конкретных потребителей.
- Не менять бинарные форматы и API модов одновременно с крупным
  рефакторингом рендера или мира.

## Почему сейчас не ECS

Текущая игра состоит из нескольких крупных уникальных объектов (`World`,
`Renderer`, игрок, стриминг) и воксельных данных, а не из тысяч сущностей
с повторяющимися компонентами. ECS добавит таблицы идентификаторов,
косвенный доступ, миграцию компонентов и сложное владение между DLL. Это
увеличит код и затруднит ABI, hot reload модов и отладку.

ECS стоит пересмотреть при появлении тысяч мобов/предметов, когда десятки
систем обходят одинаковые наборы компонентов и профиль подтверждает пользу
SoA. Тогда реестр сущностей должен быть отдельной DLL с C ABI; мир чанков,
Renderer и UI всё равно не следует переносить в ECS.
