# Мультиплеер и безопасность

## Текущий статус

Версия 0.5 содержит первый рабочий вертикальный срез разделённого
client/server runtime:

- `laiue_server.exe` — отдельный headless authoritative server без окна,
  input, renderer, mesher, content, модов и зависимости на `core`;
- `laiue_network.dll` — общий транспорт и версионированный бинарный
  протокол, используемый сервером и клиентом;
- сервер симулирует физику с частотой 60 Гц, публикует состояния с частотой
  20 Гц и единолично применяет правки блоков;
- клиент отправляет движение, взгляд и намерение редактирования, локально
  предсказывает движение и сходится к серверным координатам;
- текущий TCP listener жёстко привязан к `127.0.0.1:27180`. Его невозможно
  случайно открыть в LAN или Internet настройкой firewall.

Это фундамент мультиплеера, но ещё не готовый Internet multiplayer.
Удалённые игроки реплицируются протоколом, однако пока не рисуются. Локальный
server загружает и атомарно сохраняет `saves/default/chunks.dat`, уже прочитанный
клиентом до handshake; отдельного initial snapshot для удалённого клиента пока
нет.
Клиент делает одну попытку подключения при старте, поэтому dedicated server
следует запускать первым.

```powershell
cmake --preset ninja-clang-release
cmake --build --preset ninja-clang-release --parallel
cd build/ninja-clang-release/bin/Release
./laiue_server.exe
# Затем запустить laiue.exe на той же машине.
```

Если сервер отсутствует или seed сохранения не совпал с серверным, клиент
остаётся в обычном offline-режиме. Мир сетевой сессии намеренно не записывается
в клиентское сохранение.

## Модель доверия

Абсолютного свойства «взломать невозможно» у сетевой игры не существует.
Пользователь контролирует свой компьютер, клиент и пакеты; внешний сервис
также может подвергаться DDoS и неизвестным уязвимостям ОС. Достижимая цель —
компрометация клиента не должна давать право менять серверное состояние.

| Данные/решение | Владелец | Что проверяется |
|---|---|---|
| позиция, скорость, grounded | сервер | клиент никогда их не присылает |
| коллизии и физический tick | сервер | фиксированные правила `gameplay`/`physics` |
| правка блока | сервер | луч от серверной позиции, дальность 8 блоков, тело игрока |
| направление и кнопки | клиент как intent | конечность, диапазон, длина вектора, допустимые flags |
| порядок сообщений | соединение | версия, тип, точный размер, строго следующий sequence |
| нагрузка | network + server | bounded buffers, 120 input/s, 16 edit intents/s, 8 edits/s |

Сервер сбрасывает движение через 250 мс без свежего input. Неизвестный тип,
неверный размер, NaN/Infinity, невозможный вектор, повтор sequence, переполнение
очереди и сообщение не в той фазе handshake закрывают только виновное
соединение. На горячем пути нет аллокации на пакет: у peer заранее ограничены
receive/send/event buffers.

Wire-format не копирует C-структуры. Все целые поля записываются явно в
little-endian; заголовок фиксирован на 16 байтах и содержит magic, версию,
тип, payload size и sequence. Payload ограничен 64 байтами, полный frame —
80 байтами. `double` допускается только в серверном snapshot как IEEE-754 и
проверяется на конечность и предел сетевой сессии. Текущая версия протокола — 1;
несовместимый клиент 0.5 не проходит `CLIENT_HELLO`.

## Защита процесса

Все EXE и DLL теперь собираются с `/GS` и инициализируют собственный security
cookie до первого защищённого вызова. Для бинарников также включены ASLR,
high-entropy ASLR, DEP/NX, Control Flow Guard и CET compatibility. Это defence
in depth, а не замена проверкам входных данных.

## Что обязательно до удалённых подключений

1. Реализовать второй backend `network.dll` на
   [MsQuic](https://github.com/microsoft/msquic): TLS 1.3 с обязательной
   проверкой сертификата сервера. Не создавать собственное шифрование и не
   разрешать plaintext fallback. MsQuic шифрует пакеты, аутентифицирует
   handshake и предоставляет congestion control, streams и datagrams.
2. Не применять gameplay-команды из QUIC 0-RTT: они допускают replay.
   До завершения handshake разрешать только идемпотентное согласование версии.
3. Добавить identity/session layer: короткоживущий подписанный access token,
   привязка account → connection, ротация ключей, отзыв сессии. Пароли не
   должны передаваться игровому серверу.
4. Разделить каналы: надёжный ordered stream для handshake/world deltas;
   QUIC datagrams с tick/sequence для input и snapshots, где устаревший пакет
   можно выбросить. Ограничить idle/handshake timeouts и flow-control windows.
5. Добавить interest management по чанкам, initial snapshot, revision каждого
   чанка и упорядоченные дельты. Сжатие включать только после лимита
   распакованного размера и бюджета CPU, иначе оно создаёт zip-bomb/DoS риск.
6. Выделить серверные сохранения и реализовать конфигурацию
   адреса/порта/сертификата,
   graceful shutdown, аудит административных команд и метрики отказов без
   записи токенов и персональных данных в лог.
7. Перед открытием порта наружу нужны fuzzing protocol decoder, soak/load и
   негативные проверки malformed traffic. По решению проекта автоматические
   тесты сейчас отложены, поэтому Internet listener остаётся заблокированным.

MsQuic документирует лимиты handshake/idle, flow control и datagram support в
[официальных настройках](https://github.com/microsoft/msquic/blob/main/docs/Settings.md).
Даже после перехода на QUIC защита от объёмного DDoS требует внешнего
rate-limiting/firewall и не решается одним игровым процессом.
