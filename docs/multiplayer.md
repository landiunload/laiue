# Мультиплеер, моды и безопасность

## Текущий статус

Версия 0.5 содержит локальный вертикальный срез client/server runtime:

- `laiue_server.exe` — отдельный headless authoritative server;
- `laiue_network.dll` — общий bounded transport и бинарный протокол v4;
- сервер симулирует физику с частотой 60 Гц, публикует состояния с частотой
  20 Гц и единолично применяет правки блоков, инвентарь и подбор дропов;
- клиент отправляет input/edit intent, локально предсказывает движение и
  сходится к серверным координатам;
- сервер загружает `server`/`both` моды из `mods/server_enabled.txt`;
- клиент входит только с точным серверным набором модов; чисто клиентские
  моды не участвуют в сравнении;
- главное меню разделяет одиночную игру со списком миров и сетевую игру со
  списком из `servers.txt`.

Текущий listener жёстко привязан к `127.0.0.1`. Записи `servers.txt` уже
отделены от UI и поддерживают разные локальные порты, но удалённый адрес
намеренно отклоняется, пока нет TLS 1.3 и аутентификации сервера.

```powershell
cmake --preset ninja-clang-release
cmake --build --preset ninja-clang-release --parallel
cd build/ninja-clang-release/bin/Release
./laiue_server.exe
./laiue.exe
```

Клиент больше не подключается автоматически: выберите «Сетевая игра» и
сервер. Параметры dedicated server находятся в `server.cfg`:

```text
port = 27180
maximum_peers = 16
world_seed = 42
allow_content_downloads = false
```

Для нескольких локальных инстансов порт и разрешение загрузок можно
переопределить переменными `LAIUE_SERVER_PORT` и
`LAIUE_SERVER_ALLOW_CONTENT_DOWNLOADS`.

## Handshake модов

```text
CLIENT_HELLO
    <─ SERVER_MOD_LIST + SERVER_MOD_ENTRY[]
       клиент сравнивает id + version + SHA-256 в серверном порядке
       ├─ точное совпадение: CLIENT_MOD_LIST + CLIENT_MOD_ENTRY[]
       ├─ всё установлено: явное предложение включить нужный состав
       └─ чего-то нет: явное предложение скачать, только если сервер разрешил
    <─ SERVER_WELCOME или SERVER_REJECT
```

До `SERVER_WELCOME` сервер не создаёт игрового игрока и не принимает input.
Лишний включённый `server`/`both` мод считается несовпадением так же, как
отсутствующий. `client`-моды исключены из набора, остаются включёнными при
переключении серверного профиля и не ограничены по количеству в пределах
локального лимита каталога.

Порядок является частью совместимости: библиотечный мод должен находиться
раньше потребителя. Одинаковые `id/version`, но другая DLL не совпадут, потому
что отпечаток включает манифест и entry DLL.

## Загрузка содержимого

При `allow_content_downloads = true` сервер один раз собирает bounded bundle:

- обязательные серверные `.lmp`;
- доступные серверу `.lsp`;
- доступные серверу `.ltp`.

Клиент начинает передачу только после нажатия «Загрузить содержимое и войти».
Поток ограничен 256 МиБ и 4096 файлами, имеет SHA-256, запрещает абсолютные
пути, `..`, reparse points и неизвестные типы. Весь bundle проверяется до
записи. Файлы распаковываются в `*.download`; прежний пак переименовывается в
`*.previous`, и только затем staging атомарно становится рабочим паком.
Нативные моды выгружаются до подмены DLL, каталог перечитывается после неё,
а клиент ещё раз строит и предъявляет точный набор серверу.

SHA-256 гарантирует целостность полученного от данного соединения потока, но
не делает код сервера доверенным. Нативный мод имеет права процесса клиента.
Поэтому нет фоновой автоустановки или автоисполнения: подключаться и скачивать
следует только с доверенного сервера. Для Internet transport сертификат TLS
должен аутентифицировать источник bundle.

## Модель доверия

Абсолютного свойства «взломать невозможно» у сетевой игры не существует:
пользователь контролирует свой компьютер и клиент, а внешний сервис может
подвергаться DDoS и неизвестным уязвимостям ОС. Реализуемая гарантия —
скомпрометированный клиент не получает права напрямую менять серверное
состояние.

| Данные/решение | Владелец | Что проверяется |
|---|---|---|
| позиция, скорость, grounded | сервер | клиент никогда их не присылает |
| коллизии и fixed tick | сервер | общие правила `gameplay`/`physics` |
| правка блока | сервер | луч от серверной позиции, дальность и тело игрока |
| ломание | сервер | неизменная цель, минимальное время материала и timeout intent |
| установка | сервер | выбранный слот, тип блока, наличие и атомарный расход предмета |
| дроп/подбор | сервер | id предмета, задержка, дистанция и свободное место инвентаря |
| направление и кнопки | клиент как intent | конечность, диапазон, допустимые flags |
| моды | серверная политика | точные id/version/SHA-256 и порядок до допуска |
| wire-поток | network | magic, версия, exact size, sequence и state machine |
| нагрузка | network + server | bounded buffers, rate limits и timeouts |

Wire-format не копирует C-структуры. Заголовок фиксирован на 16 байтах и
содержит magic, версию, тип, payload size и sequence. Обычные игровые payload
остаются компактными (полный инвентарь — 109 байт); максимум 1024 байта нужен только chunks передачи
содержимого. Неизвестный тип, неверный размер, NaN/Infinity, повтор sequence,
переполнение очереди или сообщение не в той фазе закрывают только виновное
соединение.

## Профиль процесса

По решению владельца EXE и DLL собираются с `/GS-`, без CFG и CET
compatibility. Сохранены ASLR, high-entropy ASLR и DEP/NX. Протокольные
проверки и SHA-256 не отключены: они являются частью корректности входных
данных, а не общим compiler hardening.

Это осознанный компромисс. При неизвестной ошибке памяти нет compiler-level
барьеров `/GS`/CFG, поэтому plaintext listener остаётся loopback-only.

## Что обязательно до удалённых подключений

1. Второй backend на MsQuic: TLS 1.3, обязательная проверка сертификата,
   никакого plaintext fallback. Gameplay-команды не применять из QUIC 0-RTT.
2. Identity/session layer с короткоживущими подписанными токенами, ротацией
   ключей и отзывом сессий; пароль не передаётся игровому серверу.
3. Ordered stream для handshake/world deltas и QUIC datagrams с tick/sequence
   для input/snapshots.
4. Interest management по чанкам, initial snapshot, revision чанков и
   ограниченное по распакованному размеру сжатие.
5. Отдельные серверные сохранения/миры, graceful shutdown, аудит команд и
   метрики отказов без секретов в логах.
6. Перед открытием порта наружу — fuzzing decoder, soak/load и негативные
   проверки malformed traffic. Автоматические тесты пока отложены решением
   проекта, поэтому Internet listener разблокировать нельзя.

Даже после QUIC защита от объёмного DDoS требует внешнего rate limiting и не
решается одним игровым процессом.
