# Архитектура подключаемого содержимого

**Статус:** предложено (код не менялся)
**Дата:** 2026-07-16
**Область:** моды, текстурпаки, шейдерпаки, «ресурсы», «данные» —
что оставить, что убрать, как им жить дальше.

## Контекст

Сейчас в движке пять категорий содержимого (см.
[content_formats.md](content_formats.md)) в разной степени готовности:

| Категория | Формат | Состояние |
|---|---|---|
| Текстурпаки | `.ltp` (LTP1/LTP2) | работает: загрузка, fallback, переключение в меню |
| Шейдерпаки | `.lsp` (каталог `.ls`) | работает: подмена байткодов всех стадий, активация на лету |
| Моды | `.lm` | только формат и примеры; системы нет |
| Данные | `.ld` | только формат и примеры; системы нет |
| Ресурсы | `.lr` | только формат и примеры; системы нет |

Общая инфраструктура уже правильная: единый каталог
(`src/content/content_catalog.*`) с перечислением, активацией через
`active.txt` и проверкой имён; вкладка «Паки» в меню; авторинг вне игры
(fxc, PowerShell-пайплайн).

Силы, определяющие решения:

1. Движок без CRT: никаких тяжёлых рантаймов (интерпретаторы, zip,
   JSON-парсеры) без крайней необходимости.
2. Битое содержимое не валит игру: любой сбой — тихий откат на
   встроенное (так уже ведут себя текстуры и шейдеры).
3. Принцип из [architecture.md](architecture.md): API появляется только
   при наличии реальных производителей и потребителей. Категория без
   работающего потребителя — мёртвый вес.
4. Паки эксклюзивны (один активный), моды по природе композитны
   (несколько одновременно, важен порядок).

## Решение 1: три категории вместо пяти

Пользовательскими остаются **текстурпаки** (вид блоков), **шейдерпаки**
(вид рендера) и **моды** (поведение). Категории «ресурсы» и «данные»
как самостоятельные активируемые сущности упраздняются.

### Почему `.lr` («универсальные ресурсы») не нужны

Категория не имеет собственной семантики: «универсальный ресурс» — это
всегда либо данные для встроенной системы, либо часть мода. Оба текущих
примера это подтверждают: `sky_gradients.lr` — данные системы неба,
`palette_soft.lr` — вход авторского генератора текстур (вообще не
рантайм-сущность). Граница между `.lr` и `.ld` не формулируется одним
предложением — верный признак лишней категории. Файлы переезжают в
`.ld`, тип удаляется из каталога.

Ресурспак в смысле майнкрафта (текстуры + звуки + шрифты + локализация
одним контейнером) — не пятая категория, а будущая эволюция `.ltp`,
когда в движке появится звук. До этого момента вводить его — YAGNI.

### Почему датапаки не нужны (сейчас)

В майнкрафте датапак управляет контентом, которого у laiue нет:
рецепты, лут, структуры, функции, advancement'ы. Наши кандидаты в
«данные» другие, и у каждого свой правильный дом:

- пресеты генерации мира (`world_presets.ld`) — это выбор при создании
  мира; своё место они получат на экране создания мира вместе с
  сохранениями, а не в списке «активных паков»;
- свойства блоков, градиенты неба — параметры движка; когда их станет
  нужно менять извне, их публикует реестр параметров модов (решение 2);
- всё остальное «дай движку табличку» — полезная нагрузка мода.

Итог: `.ld` **остаётся форматом файла** (построчный `LAIUE DATA`), но
перестаёт быть категорией с `active.txt` и вкладкой. Каталог `data/`
хранит данные движка; моды возят свои данные внутри себя.

## Решение 2: модель модов — два уровня

### Сравнение вариантов исполнения

| Вариант | Сложность | Мощность | Безопасность | Совместимость с «без CRT» |
|---|---|---|---|---|
| A. Декларативные параметры | низкая | твики опубликованного | полная | идеальная |
| B. Нативные DLL с C ABI | средняя | полная | нулевая (полное доверие) | идеальная — модель DLL уже есть |
| C. Скрипты (Lua) | высокая | высокая | средняя | плохая: порт setjmp/аллокаторов/stdio |
| D. WASM-интерпретатор (wasm3) | высокая | высокая | высокая | средняя; медленный интерпретатор |

**Принято: A сейчас, B — следующим уровнем, C/D — отвергнуты** (C —
чужеродна философии движка; D — вернуться, только если понадобится
запуск недоверенных модов).

### Уровень A: декларативные моды (`.lm`) — ближайшая цель

Мод — набор переопределений **опубликованных параметров движка**.
Движок объявляет реестр параметров: имя (`gameplay.extra_air_jumps`),
тип, диапазон (значения клампятся), момент применения — «на лету» или
«со следующим миром». Неизвестные ключи игнорируются: старый движок
спокойно ест новый мод.

Композиция вместо единственного `active.txt`: `mods/enabled.txt` —
упорядоченный список имён, по строке на мод. Конфликт ключей решается
просто: последний в списке побеждает. Это единственное расширение,
которое требуется каталогу.

Первые параметры реестра выбраны так, чтобы оба существующих
примера заработали по-настоящему, а не остались декорацией:

| Параметр | Потребитель | Применение |
|---|---|---|
| `gameplay.extra_air_jumps`, `gameplay.air_jump_impulse` | player_jump | на лету |
| `time.locked`, `time.day_length_minutes` | game_time | на лету |
| `world.preset` (ссылка на `data/world_presets.ld`) | генерация | со следующим миром |

### Уровень B: нативные DLL-моды — после сохранений

Форма: каталог `mods/<имя>.lmp` с манифестом `mod.lm`
(`entry = mod.dll`, `api = 1`) и самой DLL. Контракт:

- единственный экспорт `LaiueModMain(const LaiueEngineApi* api)`;
- `LaiueEngineApi` — версионируемая таблица функций: первые поля
  `uint32 sizeBytes` и `uint32 version`; расширение — только
  добавлением полей в хвост; несовпадение мажора — отказ загрузки;
- в API входят: чтение/запись блоков (с автоинвалидацией чанков),
  время, регистрация хуков (`onFrame`, `onBlockEdit`, `onPlayerTick`);
- все хуки — только главный поток; воркеры мешера мод не видит;
- доверие полное, как у Forge: DLL — это движок. Песочницы нет,
  и это фиксируется честно, а не обещается.

Уровень B сознательно отложен: по принципу из architecture.md события
(`game_events.h`) проектируются, когда есть несколько реальных
потребителей. Ими станут DLL-моды — тогда и появится API.

## Решение 3: контракт шейдерпака

Сегодня `.lsp` обязан молча совпасть с root-сигнатурами и раскладкой
cbuffer'ов движка и предоставить все шесть байткодов. Два дополнения:

1. Манифест `pack.lm` (`LAIUE SHADER 1`) с полем `contract = N` —
   версией шейдерного контракта движка. Контракт — задокументированный
   перечень стадий, сигнатур, cbuffer-раскладок и соглашений (sRGB,
   гамма). Несовпадение — пак виден в списке, но помечен несовместимым
   и не активируется; при старте с несовместимым активным — откат на
   встроенные.
2. Частичные паки: отсутствующий `.ls` означает встроенный байткод этой
   стадии. Пак «только `chunk_ps.ls`» становится законным — порог входа
   для авторов резко ниже.

## Итоговая схема

```text
пользовательское содержимое      discovery/активация         потребители
textures/*.ltp   ─┐
shaders/*.lsp    ─┼─  content_catalog  ─┬─>  render: текстуры, шейдеры
mods/*.lm|*.lmp  ─┘   active.txt        └─>  core/gameplay: реестр
                      enabled.txt (моды)      параметров, позже DLL-хуки
data/*.ld  — данные движка (пресеты мира и т.п.), без активации
```

## Чего не делаем (зафиксированные отказы)

- **Zip-контейнеры** — без CRT нет распаковщика, каталоги достаточны.
- **Скрипты** — см. решение 2.
- **Hot-reload DLL-модов** — сложность фенсов/состояния не окупается.
- **Подписи, маркетплейс, сетевые протоколы** — вне горизонта.
- **Пятая категория «ресурсы»** — упразднена решением 1.

## Последствия

Проще: одна ментальная модель («паки — вид, моды — поведение, данные —
внутренности»), меньше каталогов и вкладок, оба примера модов получают
реальную реализацию малой кровью.

Сложнее: переезд `.lr` → `.ld` (правка content_catalog и примеров),
каталогу нужен режим упорядоченного списка. Пересмотреть при появлении
звука (ресурспак-контейнер) и сохранений (экран создания мира, DLL-API).
